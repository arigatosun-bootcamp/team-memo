# Day 5: バグ修正研修ガイド

## 概要

Day 5では、実際の開発現場で遭遇する**バグ修正**を体験します。
「TeamMemo」というチームメモ共有アプリに報告された5つのバグを、
GitHub Issueに基づいて順番に修正していきます。

## 目標

- 曖昧なバグ報告から原因を特定する力を身につける
- AIツールを活用しつつ、自分の頭で考えてバグを修正する
- 再発防止テストを書く習慣を身につける
- Git/PRワークフローに沿った修正プロセスを体験する

## 環境セットアップ

```bash
# 1. リポジトリをクローン
git clone <リポジトリURL>
cd template-c-team-memo

# 2. 依存パッケージをインストール
npm install

# 3. 環境変数を設定
cp .env.example .env
# .env を編集してSupabaseの情報を入力

# 4. 開発サーバーを起動
npm run dev

# 5. テストが通ることを確認
npm test
```

## 各Issueの進め方

### 手順

1. **Issueを読む** — 曖昧な報告を注意深く読み、何が問題かを理解する
2. **ブランチを切る** — `fix/issue-{番号}-{説明}` 形式
3. **再現する** — 実際にアプリを操作して問題を再現する
4. **原因仮説を立てる** — 最低2〜3個の仮説を考える
5. **AIツールで調査・修正する** — Claude Codeを活用する
6. **再発防止テストを書く** — バグが再発しないテストを追加
7. **PRを作成する** — テンプレートに沿って記入

### 重要な心構え

- **AIの回答を鵜呑みにしない** — AIが「問題ない」と言っても、自分で確認する
- **テストが通っている ≠ バグがない** — テスト自体が不十分かもしれない
- **コメントを信頼しすぎない** — コードのコメントが正しいとは限らない
- **型定義を信頼しすぎない** — TypeScriptは実行時の型を保証しない
