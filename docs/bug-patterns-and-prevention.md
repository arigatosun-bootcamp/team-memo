# バグパターンと予防策

このドキュメントは、Team Memo プロジェクトで発見・修正されたバグのパターンと、今後の予防策をまとめたものです。

---

## 1. いいねボタンのフリーズ（Issue #4 / PR #25相当）

**パターン**: `isLoading` を `finally` でリセットしない設計で、エラー時にボタンが操作不能になる

**原因**: いいねは1回限りの操作のため、成功時は `isLoading` リセット不要と判断。しかしネットワークエラー時に `isLoading` が `true` のままになり、ボタンがフリーズする。

**修正**: catch ブロックで楽観的更新を巻き戻す処理は維持しつつ、エラー時も `isLoading` を適切にリセット。

**予防策**: 非同期処理では `finally` ブロックでローディング状態をリセットするのが基本。例外的に省略する場合はエラーケースを必ず検証する。

---

## 2. 自己通知の発生（Issue #10）

**パターン**: 通知作成時に「自分自身へのアクション」を除外していない

**原因**: `createNotification()` で `userId === actorId` のチェックがなく、自分のメモにいいねすると自分に通知が届く。

**修正**: `createNotification()` の先頭に `if (userId === actorId) return;` を追加。

**予防策**: 通知・メール送信などのアクション系関数では、「自分自身への操作」を常にフィルタリングする。

---

## 3. 通知ベルのポーリング問題（Issue #10）

**パターン**: useEffect の依存配列と関数定義の不整合

**原因**: `fetchUnreadCount` が useEffect の外で定義されており、依存配列の警告を抑制するために eslint-disable を使用。結果、ポーリングが正しく動作しない。

**修正**: `fetchUnreadCount` を useEffect 内に移動し、依存配列を `[userId]` に修正。

**予防策**: `eslint-disable` でフック警告を抑制するのは避ける。警告が出たら設計を見直す。

---

## 4. プロフィール更新が反映されない（Issue #11）

**パターン**: データの更新が一箇所のみで、関連するデータソースが同期されない

**原因**: profiles テーブルは更新するが、auth.users の user_metadata を更新していない。また、更新後のレスポンスでフロントエンドの状態を更新していない。

**修正**: `supabase.auth.admin.updateUserById()` で auth metadata も同期。フロントエンドで `setUserName()` を呼び出し。

**予防策**: データが複数テーブル/ストアにまたがる場合、すべての箇所を同期する。更新APIのレスポンスでフロントエンドの状態も更新する。

---

## 5. ダッシュボードランキングの表示不具合（Issue #12）

**パターン**: JOINに必要な外部キーの欠如 + 非公開メモのフィルタリング漏れ

**原因**:
- memos テーブルから profiles テーブルへの外部キーが未設定で、JOIN がエラーになる
- ランキングに `.eq("is_private", false)` のフィルタがなく、非公開メモも含まれる

**修正**: FK制約 `memos_user_id_profiles_fkey` を追加。ランキングクエリに非公開フィルタを追加。

**予防策**: Supabase の JOIN は外部キーに依存するため、テーブル設計時に必要な FK を確認する。公開データのクエリでは `is_private` フィルタを忘れない。

---

## 6. タイムゾーンによる日付グラフのずれ（Issue #12）

**パターン**: サーバー側の日付処理でタイムゾーンを指定していない

**原因**: `toLocaleDateString("ja-JP")` はサーバーのタイムゾーンに依存する。UTC サーバーでは日本時間の深夜のメモが前日としてカウントされる。

**修正**: `toLocaleDateString("ja-JP", { timeZone: "Asia/Tokyo" })` でタイムゾーンを明示。

**予防策**: 日付処理では常にタイムゾーンを明示的に指定する。特にサーバーサイドレンダリングではサーバーの TZ に依存しないコードにする。

---

## 7. タグの大文字小文字重複（Issue #13）

**パターン**: 文字列比較で大文字小文字を区別してしまう

**原因**: タグ入力時の重複チェックが `includes()` のみで、"React" と "react" を別タグとして扱う。また、DB の UNIQUE 制約違反（23505）時にエラーハンドリングがない。

**修正**: `.toLowerCase()` での大文字小文字無視比較。UNIQUE 制約エラー時は既存タグを返すように変更。

**予防策**: ユーザー入力の文字列比較では大文字小文字を正規化する。DB の UNIQUE 制約エラーは 500 ではなく適切にハンドリングする。

---

## 8. 管理者API認可の欠如（Issue #14）

**パターン**: ミドルウェアの matcher 漏れ + API ルートの直接保護なし

**原因**:
- middleware の matcher が `/admin/:path*` のみで `/api/admin/:path*` を含んでいない
- API ルート自体にも認可チェックがない（多層防御の欠如）
- ミドルウェアが API リクエストに対して HTML リダイレクトを返してしまう

**修正**: matcher に `/api/admin/:path*` を追加。API ルートに `verifyAdmin()` ヘルパーを追加。API リクエストには JSON エラーを返すように分岐。

**予防策**: API ルートは必ず自身で認可チェックを行う（多層防御）。ミドルウェアでは API リクエストと画面リクエストのレスポンス形式を分ける。

---

## 9. いいね・ブックマークAPIのなりすまし（PR #30）

**パターン**: リクエストボディの `user_id` を信頼している

**原因**: like API が `body.user_id` をそのまま使用しており、任意のユーザーIDを指定してなりすましが可能。bookmark API も同様。

**修正**: Authorization ヘッダーのトークンから `supabase.auth.getUser(token)` でユーザーを検証する方式に変更。

**予防策**: ユーザー識別情報はクライアントから送信された値を信頼せず、必ずサーバー側でトークン検証する。`supabaseAdmin` は RLS をバイパスするため、アプリケーションコードで認可を実装する。

---

## 10. Bookmark 型定義の null 未考慮（PR #31）

**パターン**: 外部キー参照先のデータが null になりうるケースを型で表現していない

**原因**: `Bookmark.memo` が `Memo` 型だが、RLS により非公開メモが null として返される場合がある。ブックマーク一覧で `bookmark.memo.title` にアクセスするとランタイムエラー。

**修正**: `memo: Memo | null` に変更し、オプショナルチェイニング (`?.`) で安全にアクセス。

**予防策**: RLS が有効なテーブルの JOIN 結果は null になりうることを型で表現する。TypeScript の strict null check を活用する。

---

## 11. いいねカウントの非アトミック更新（PR #31）

**パターン**: SELECT → 計算 → UPDATE の3ステップで競合状態が発生する

**原因**: 複数ユーザーが同時にいいねすると、SELECT で取得した値が古くなり、カウントが正しく増加しない（ロストアップデート）。

**修正**: `increment_likes_count` RPC 関数を作成し、`UPDATE memos SET likes_count = likes_count + 1` のアトミック操作に変更。

**予防策**: カウンター更新は必ず DB 側でアトミックに行う。`SET column = column + 1` パターンまたは RPC 関数を使用する。

---

## 12. isAdmin のデッドコード（PR #31）

**パターン**: 存在しない値との比較条件がコードに残っている

**原因**: `data?.role === "owner"` の条件があるが、profiles テーブルの role は `"member" | "admin"` のみで `"owner"` は存在しない。害はないが混乱の原因になる。

**修正**: `"owner"` 条件を削除。

**予防策**: 型定義と実際のデータスキーマを一致させる。TypeScript の union 型を活用して、存在しない値との比較をコンパイル時に検出する。
